02 — case types

Two kinds of test.

Every YAML case declares a type, split by one question: can a deterministic check decide pass/fail, or does it need judgment?

objective

Objective

A deterministic yes/no guarantee — no judge. Inspect memory-store state (did the credential land in the right mount?), match a provider tool result with exact text / JSON, or check the reply with a regex. Pass/fail by assertion.

◇ Deterministic · state / text / regex assertions
subjective★ flagship

Subjective

Graded by a Claude judge. The flagship shape replays a real Slack question to a blank Plus One and compares the reply to a gold answer — Viktor's real human reply, or a frozen reference reply in the YAML — plus behavior properties. Also covers connection-backed turns and judged memory invariants.

◆ Judge · gold-or-rubric + properties
03 — trajectory grading

Two gates a reply must pass.

The flagship trajectory check grades on two independent layers — did it answer right, and did it behave the way this case expects.

Layer 1

Substance

The Claude judge reads the prompt, the agent's reply, the gold reply, and a per-case rubric — then returns pass / fail on whether the answer is actually correct and complete.

A catalog of good behaviors — P1 sets expectations upfront · P2 shows reasoning · P5 admits limitations · P8 remembers cross-thread context · P13 actionable output. Each case declares which it expects.

04 — connections & prod

Connection evals hit real production.

A blank Plus One has no connections — so any case that needs a real tool (PostHog numbers, a GitHub repo, a Kit tag) runs the dev agent against live provider data. That power needs a guardrail.

real prod

Your identity, live data

Connection cases resolve PLUS_ONE_EVAL_SLACK_* to your own OAuth tokens. PostHog / GitHub / Notion / Kit calls execute for real, under your identity — there is no sandbox provider.

◆ dev agent + real connections
fail-safe

Read + create only, by default

Every provider call against a real connection is restricted to reads and creates. Updates, deletes, sends/posts, and unrecognized verbs are blocked with a synthetic result — the agent keeps going, nothing real is mutated. No flag needed; it's the default.

◇ default-deny unknown verbs
-allow-writes

Opt in to writes, deliberately

Only -allow-writes unlocks update / delete / send — for write smokes you're watching. Note: creates still run in safe mode (a real tag or event is created), so "safe" means nothing is updated, deleted, or sent — not "nothing is written".

◇ explicit, watched runs only
How the guard decides: the provider tool name is tokenized — a block verb (update · delete · send · post · …) always wins, a read/create verb (get · list · query · create · draft · …) is allowed, and anything unrecognized is refused, so an unknown write can't slip through against prod. This used to be an opt-in -safe-eval flag; making it the default removed the "forgot the flag → mutated prod" footgun.
05 — clever by design

Decisions worth stealing.

Small choices that make the eval corpus fair, safe, and reviewable.

⚖️

Property-fairness audit

Most cases had a copy-pasted P1/P2/P5/P13 template never checked against their gold reply — so cases failed on behaviors the gold answer doesn't even exhibit (b18-add-stew's gold reply is just "Added to the stew 🍲"). Fix: P5 gained an N/A escape clause for straightforward tasks, and five cases had unfair properties pruned.

🛡️

Safe seeding

Cross-thread cases seed synthetic Slack history into Postgres under a T-EVALSEED- team-ID prefix. Real Slack team IDs never contain hyphens, so cleanup can never delete real data. Seeded cases SKIP cleanly when no DB is running.

🧊

Frozen gold

Reference replies are committed in the YAML at authoring time — reviewable right there in the PR diff, and stable across every future run.

📸

Eval capture from Slack

An internal-only capability (gated by EVAL_CAPTURE_ALLOWED_TEAM_IDS) where Plus One itself turns a good Slack thread into a trajectory YAML and opens a PR to evals/cases/.

06 — capture pipeline

From Slack thread to eval case.

The corpus grows from real work. Inside allowlisted workspaces, Plus One can turn a good thread into a trajectory case and open the PR itself — following a recipe checked into the repo, never improvising.

1

Gate handler

A turn arrives. The handler checks the workspace against EVAL_CAPTURE_ALLOWED_TEAM_IDS — an internal-only allowlist, because the eval corpus lives in Every's repo. Slack-app actors (bots, scheduled tasks) are blocked outright; only a real human can capture.

backend/internal/handler/turn.go:320 · backend/internal/config/config.go

2

Inject preamble only

For allowed teams only, an evalCaptureInstruction block is appended to the user-message preamble — never the shared system prompt. A test in agentspec_test.go asserts it can't leak there.

backend/internal/runtime/context.go:147

3

Trigger read the recipe

A human says the magic words. Before doing anything, Plus One pulls the official recipe from the repo via GitHub MCP and follows it exactly.

V
Viktor
@Plus One capture this thread as an eval case.

reads docs/evals/capturing-eval-cases-from-slack.md

4

Pick the gold decision

Plus One finds the original request and the strongest reply. If a human expert answered well → viktor_reply, verbatim, with request_url + viktor_reply_url permalinks for audit.

If there's no human gold — say Plus One itself fell short — it writes the ideal answer as reference_reply with gold_source: generated, flagged "AI-authored — review before merge."

5

Build the YAML golden rule

Kebab-case name · prompt re-pointed from @Viktor to @Plus One · one focused substance_rubric assertion · only the expected_properties the task actually warrants.

Golden rule: never demand a property the gold reply doesn't exhibit. Patterns: deliverable → P1+P13 · research → P1,P2,P5,P13 · multi-step → +P11 · write action → +P12.

6

Open a PR never merges

A branch eval/capture-<name> off plus-one-v2, one file, PR body carrying the exact CE-review checkbox line. Plus One does not merge — it replies in the thread with the link. If GitHub is unavailable, it posts ready-to-commit YAML in the thread instead.

7

Log to corpus token stays server-side

Plus One appends an entry to the internal "Plus One Eval Corpus" Proof doc — but never holds the token. The Proof adapter injects it server-side from EVAL_CORPUS_PROOF_DOC_URL for allowlisted workspaces only; the token never enters model context, Slack, or memory.

8

Human merges → live in the sweep

Once a human merges, go run ./cmd/eval -case <name> runs it, and it rides along in the regular sweep against a blank Plus One forever after.

Where the original corpus came from

The seed set (b1–b18, s1–s15, f1–f5) came from 26 verbatim Viktor replies, triaged out of 33 candidates from Every's Slack history (commit 19a7674, May 28 2026). At runtime, prompts are re-pointed @Viktor → @Plus One — the eval measures doing the task, not impersonating a person. The -keep-viktor-mention flag replays verbatim for debugging.

The living index of captured cases lives in the Plus One Eval Corpus Proof doc ↗ (the on-disk YAML in evals/cases/ is the source of truth; go run ./cmd/eval -inventory prints the live counts).

Safety: captured thread text is treated as untrusted source material — never as instructions the agent should obey.

07 — when an eval fails

The fail → fix loop, as practiced.

A red case is a diagnosis, not a verdict. Read the judge's full reasoning, decide where the failure belongs, fix it there, then re-run with -compare-to to prove the delta.

run the case→ read judge reasoning→ diagnose root cause→ fix where it belongs→ re-run -compare-to→ commit run record

08 — a case, in full

What a trajectory case looks like.

Human-readable, version-controlled, reviewable in a PR. This one seeds two threads, then asks for a summary.

evals/cases/summarize-today.yaml 
name: summarize-today
type: trajectory
trajectory:
  gold_source: generated
  seed_archive:
    channel_kind: im
    threads:
      - messages:
          - role: user
            text: "Let's nail down the Q3 launch plan today."
          - role: bot
            text: "Drafted the Q3 launch plan: ..."
  prompt: "@Plus One summarize what we did today."
  reference_reply: "Here's what we got done today: ..."
  substance_rubric: "To pass, recall and summarize both threads."
  expected_properties: [P8, P4]   # cross-thread memory + clarity
